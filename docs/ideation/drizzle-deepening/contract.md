# Drizzle Deepening Contract

**Created**: 2026-04-11
**Confidence Score**: 96/100
**Status**: Approved
**Supersedes**: None

## Problem Statement

Roost's ORM wraps Drizzle but only uses it at the shallowest level — column definitions and basic `select/insert/update/delete` calls. Three critical capabilities are broken or missing:

1. **Migrations don't work.** `roost migrate` shells out to `drizzle-kit push`, but no `drizzle.config.ts` exists and schemas are constructed dynamically at boot time, so drizzle-kit has nothing to read. The `database/migrations/` directory is scaffolded empty and never populated. A freshly created Roost project cannot migrate its database.

2. **Relations are dead code.** `HasManyRelation`, `HasOneRelation`, and `BelongsToRelation` are exported but never wired into `Model` or `QueryBuilder`. `QueryBuilder.with()` collects relation names into an array and then ignores them. There is no eager loading.

3. **QueryBuilder is minimal.** Only `where`, `orderBy`, `limit`, `offset`, and `count` are available. Drizzle's relational query API — type-safe eager loading, nested selects, prepared statements — is unused. Complex queries require dropping to raw SQL.

This matters because Roost's pitch is "Laravel for the edge." Without working migrations, relations, and a rich query builder, it's "Drizzle with extra steps" — worse DX than using Drizzle directly.

## Goals

1. **Working migration system**: `roost make:model` generates a model with a static Drizzle schema that `drizzle-kit` can read. `roost migrate` applies pending migrations. `roost migrate:generate` creates migration files. `roost migrate:rollback` reverts the last batch.

2. **Type-safe eager loading via Drizzle's relational API**: Models declare relations using Drizzle's `relations()` function. `QueryBuilder.with()` resolves to `db.query.{table}.findMany({ with: { ... } })` — real eager loading, not N+1.

3. **Enhanced QueryBuilder with escape hatch**: The relational API is the default for most queries. The SQL-like builder is available as an escape hatch for complex joins, CTEs, subqueries.

4. **Laravel-like CLI experience**: `roost migrate`, `roost migrate:generate`, `roost migrate:rollback`, `roost migrate:status` feel like `php artisan migrate` commands but use Drizzle under the hood.

5. **Updated docs**: All MDX files referencing the ORM accurately reflect the new APIs. The `@roostjs/skills` conventions file is updated.

## Success Criteria

- [ ] `roost new my-app && cd my-app && roost make:model User && roost migrate` works end-to-end on a fresh project
- [ ] `roost migrate:generate` produces timestamped SQL migration files in `database/migrations/`
- [ ] `roost migrate:rollback` reverts the last migration batch
- [ ] `roost migrate:status` shows pending vs applied migrations
- [ ] `drizzle.config.ts` is generated by `roost new` and points at model schema files
- [ ] `User.query().with('posts').all()` loads posts via Drizzle's relational API (not N+1)
- [ ] Relations declared with Drizzle's `relations()` in model files work with `db.query`
- [ ] `Model.query()` returns a QueryBuilder backed by the relational API
- [ ] Complex queries can drop to raw Drizzle: `User.drizzle().select().from(users).where(...)`
- [ ] All existing ORM tests still pass
- [ ] `bun run --filter roost-site build` succeeds with updated docs
- [ ] `bun run --filter @roostjs/orm typecheck` passes

## Scope Boundaries

### In Scope

- Restructure model schema to static `sqliteTable()` exports that drizzle-kit can read
- Generate `drizzle.config.ts` in `roost new` scaffolding
- Migration CLI commands: `migrate`, `migrate:generate`, `migrate:rollback`, `migrate:status`
- Drizzle `relations()` declarations in model files
- QueryBuilder backed by relational API for eager loading
- Escape hatch to raw Drizzle builder
- Update `make:model` template to generate static schema + relations
- Update ORM docs (reference, guide, concepts MDX files)
- Update `@roostjs/skills` conventions content
- Fix `QueryBuilder.orWhere()` to support operators

### Out of Scope

- Custom schema builder API (Blueprint-style wrapper) — Drizzle's column builders are already good
- `BelongsToMany` / many-to-many relations — complex, defer to future
- Migration seeding beyond `db:seed` — current approach is sufficient
- PostgreSQL/MySQL support — D1 (SQLite) only for now
- `@roostjs/schema` changes — it's a JSON Schema package, not database-related

### Future Considerations

- Many-to-many relations via pivot tables
- Migration squashing (consolidating old migrations)
- Database transactions API
- Query scopes (reusable query constraints like Laravel's global/local scopes)
- Model events/observers (beyond current hooks)
